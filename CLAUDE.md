Quality facets (Definition of Done):

- Every **non-trivial** change is examined across 9 facets, not just typechecked and linted. The facets: **Security & Privacy** (auth/ownership, RLS, Zod input validation, no leaked secrets/PII), **Correctness & Invariants** (paired-file alignment, client↔edge agreement, migration safety, type drift), **Reliability & Error Handling** (the `handleError`/`HttpError` contract, failure modes, logging, offline resilience), **DB Load & Scalability** (query shape, N+1, indexes, realtime fanout, behavior at 10×), **Performance** (render cost, re-renders, bundle size, perceived latency), **Readability & Maintainability** (simplicity, reuse, no premature abstraction, file size), **Conventions & Consistency** (follows the project's established idioms so the codebase stays uniform and readable — `uniqueChannelTopic` realtime naming, `import/order` groups, feature-subfolder placement, the `jsonResponse`/`HttpError`/`parseBody` edge contracts, object-literal hook returns, big-file decomposition), **Accessibility** (labels, touch targets, contrast), and **Tests** (coverage added/updated, suite green).

Do the code/file work and leave it in the working tree; then wait. Reason: this repo lives in OneDrive and often has more than one agent/process touching it, so an unprompted `git reset`/`stash`/`checkout` can clobber another session's uncommitted WIP (this has bitten us — see the OneDrive concurrent-commit hazard). When in doubt, stop at the file edits and ask. This does NOT block the file-writing tools (Edit/Write) — only git-state mutations.

- Non-critical **post-response** work in edge functions — push fan-out (`notifyTeams`/`notifyLeague`), matview refreshes, best-effort logging — should be deferred off the response path with `deferWork(promise, label)` from [supabase/functions/_shared/background.ts](supabase/functions/_shared/background.ts). It wraps `EdgeRuntime.waitUntil` (keeps the worker alive until the promise settles) and falls back to an unawaited `.catch()` under `supabase functions serve`. `deferWork` catches + logs failures, so ONLY pass work whose failure is **non-fatal** to the request, and evaluate the promise's inputs BEFORE the call (they're captured when the promise is created). A Postgrest query builder is a thenable, not a real `Promise` — wrap it in an async IIFE: `deferWork((async () => { await supabase.rpc('…') })(), 'label')`. Deferred today: make-draft-pick, execute-trade (×3 sends), commissioner-action, create-poll, submit-seed-pick, autodraft, and import-screenshot-league's per-new-player `refresh_player_season_stats`. Do **NOT** defer: (a) work that produces state the client needs next — e.g. vote-poll's vote insert (its `23505`→409 "already voted" must surface synchronously) and submit-seed-pick's `generate-playoff-round` fetch (produces the bracket); (b) CRON / webhook / `send-notification` functions (no user is waiting on the response, so there's nothing to speed up). Caveat: `EdgeRuntime.waitUntil` keeps the worker alive until the deferred work settles, so the edge `execution_time_ms` metric may NOT drop — the real win is the client-perceived time-to-response, not server wall-time.

- Supabase realtime channel names for `postgres_changes` (and any non-shared `presence`) subscriptions created inside a `useEffect` MUST be built with `uniqueChannelTopic('base-name')` from [lib/supabase.ts](lib/supabase.ts). Plain deterministic names — and even `` `name-${Date.now()}` `` — collide when React re-fires effects within the same millisecond (notification tap, auth refresh, concurrent re-render, foreground transition) because `supabase.removeChannel()` is async. The old channel is still in `joined` state and `realtime.channel(topic)` returns it instead of creating a new one — the next `.on('postgres_changes', ...)` then throws `cannot add 'postgres_changes' callbacks ... after subscribe()` and Hermes can crash natively. `uniqueChannelTopic` combines `Date.now()` + a process-wide monotonic counter + `Math.random()` so collisions are impossible. Broadcast channels (e.g. `lottery:${leagueId}` in [app/lottery-room.tsx](app/lottery-room.tsx)) and SHARED presence channels (e.g. `chat_presence_${conversationId}` in [hooks/chat/useReadReceipts.ts](hooks/chat/useReadReceipts.ts)) are exempt — they need deterministic names so peers can match. The realtime-channel scanner ([scripts/check-realtime-channels.mjs](scripts/check-realtime-channels.mjs)) gates this in `pre-commit`.

- All edge functions return HTTP responses via [supabase/functions/_shared/http.ts](supabase/functions/_shared/http.ts) — do not hand-roll `new Response(JSON.stringify(...), { headers: ... })`. Use `jsonResponse(body, status?)` for success, `errorResponse(message, status?)` for a direct error return, and `return handleError(error, '<fn-name>')` as the *entire* body of every top-level `catch`. The contract that makes this safe: throw `new HttpError(message, status)` for **expected, client-caused** failures (validation → 400, auth → 401, forbidden → 403, not-found → 404, conflict → 409) — `HttpError`'s message IS surfaced to the client. For **anything else** (raw Postgres/Supabase errors, null derefs, upstream-API failures) throw/rethrow the error plain — `handleError` logs it and returns a generic `500 Internal server error` so DB internals never leak. Never wrap a DB error's `.message` in an `HttpError`. `_shared/cors.ts` still owns the `OPTIONS` preflight (`corsResponse()`); `http.ts` responses already include the CORS headers.

- All edge function input validation runs through [supabase/functions/_shared/validate.ts](supabase/functions/_shared/validate.ts), which re-exports Zod (`z`) and exports `parseBody(schema, body)`. Define `const Body = z.object({...})` at the top of `index.ts` immediately after the imports, then `const { ... } = parseBody(Body, await req.json())` in the handler. Validation failures throw `HttpError(400, "<path>: <message>")` so they route through `handleError` automatically — never hand-roll `if (!x) throw` for shape/type checks Zod can express. Things Zod handles natively (REMOVE the ad-hoc check): required (default), UUID (`z.string().uuid()`), string bounds (`z.string().min(N).max(M)`), number bounds (`z.number().int().min(M).max(N)`), enums (`z.enum([...])`), non-empty arrays (`z.array(...).min(1)`), optional (`.optional()`), ISO timestamps (`z.string().datetime()`), date strings (`z.string().regex(/^\d{4}-\d{2}-\d{2}$/)`), booleans. Preserve the **existing user-facing error message** when adding a Zod constraint by passing it as the second arg: `z.string().min(1, 'Question must be 1-500 characters')` — keeps client-side error strings stable. What stays inline (don't try to push to Zod): anything that depends on a DB-fetched value (cross-row checks, `closes_at > now()`, `selection_idx < options.length`), and auth/ownership checks. Discriminated union bodies (`action: 'preview' | 'execute'`) use `z.discriminatedUnion('action', [PreviewBody, ExecuteBody])` with `.passthrough()` on inner schemas only when the nested shape is genuinely free-form (Sleeper/RC payloads).

- Player names render through [components/player/PlayerName.tsx](components/player/PlayerName.tsx) everywhere they appear in a list / row / card / cell / picker. It shows the FULL name when it fits and falls back to `abbreviateFirstName` ("S. Gilgeous-Alexander" — first initial + the WHOLE last name, never an ellipsized last name) only when the full name would clip to a second line, via a hidden onTextLayout measurer. Standard usage: `<PlayerName name={x.name} type="defaultSemiBold" style={{ fontSize: ms(14) }} containerStyle={{ flexShrink: 1 }} />` — put the slot's flex/width on `containerStyle` so the measurer matches the visible width. Don't hand-render `<ThemedText>{player.name}</ThemedText>` in compact UI, and don't re-implement the measure dance (the old inline copies in `FreeAgentRow` and `matchup/PlayerCell` were consolidated into this one component). Exceptions that keep the FULL name directly: dedicated single-player **detail titles** (`PlayerDetailHeader`, the prospect detail page, single-player analytics overlays) — the name is the subject and has room; **`accessibilityLabel`/`accessibilityHint` strings** — screen readers must announce the full name; and **stringified/native-isolated surfaces** a `<View>` can't enter (text concatenated inside a parent `<Text>`, e.g. `TransactionCard`; the iOS widget's stringified layout in `widgets/MatchupActivity.tsx`) — call `abbreviateFirstName(name)` directly there instead.
