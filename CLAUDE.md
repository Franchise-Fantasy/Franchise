- Supabase realtime channel names for `postgres_changes` (and any non-shared `presence`) subscriptions created inside a `useEffect` MUST be built with `uniqueChannelTopic('base-name')` from [lib/supabase.ts](lib/supabase.ts). Plain deterministic names — and even `` `name-${Date.now()}` `` — collide when React re-fires effects within the same millisecond (notification tap, auth refresh, concurrent re-render, foreground transition) because `supabase.removeChannel()` is async. The old channel is still in `joined` state and `realtime.channel(topic)` returns it instead of creating a new one — the next `.on('postgres_changes', ...)` then throws `cannot add 'postgres_changes' callbacks ... after subscribe()` and Hermes can crash natively. `uniqueChannelTopic` combines `Date.now()` + a process-wide monotonic counter + `Math.random()` so collisions are impossible. Broadcast channels (e.g. `lottery:${leagueId}` in [app/lottery-room.tsx](app/lottery-room.tsx)) and SHARED presence channels (e.g. `chat_presence_${conversationId}` in [hooks/chat/useReadReceipts.ts](hooks/chat/useReadReceipts.ts)) are exempt — they need deterministic names so peers can match. The realtime-channel scanner ([scripts/check-realtime-channels.mjs](scripts/check-realtime-channels.mjs)) gates this in `pre-commit`.

- Code organization conventions enforced by ESLint and the Phase 1/2 reorg:
  - **Imports**: `import/order` rule auto-sorts groups (builtin → external → `@/` internal → relative) with a blank line between groups and alphabetized within. Don't fight the auto-fix; if a stale comment block traps imports below it, move the comment, not the imports.
  - **`components/` root**: don't add new files here. Use a feature subfolder (`account/`, `banners/`, `ui/`, `player/`, `roster/`, etc.) — pick one that already exists or matches an existing theme.
  - **`utils/` root**: same — group by domain (`utils/roster/`, `utils/scoring/`, `utils/nba/`, `utils/league/`). The root keeps only truly cross-cutting primitives (`buildConfig`, `dates`, `formatting`, `network`, `scale`, `moderation`, `sleeperMapping`).
  - **Inline Props types**: fine at the top of a component file. Only move to `types/` if reused across multiple files — duplication is cheaper than premature abstraction here.
  - **Hook return shape**: object literal `{ data, isLoading, error }`, not tuples — exception is the `useState`-style `[state, setter]` pattern.
  - **Hook size**: if a hook exceeds ~300 lines, split by data source rather than letting one hook own multiple queries plus view state.
  - **Big-file decomposition pattern** (used in `PlayerDetailModal`, `roster.tsx`, `FreeAgentList`, `matchup.tsx`): when a file passes ~1500 lines, extract first the StyleSheet to `<name>Styles.ts`, then any inline sub-components (skeletons, modals, banners) defined above the main component, then any cohesive view-section sub-trees inside the main render. Don't try to lift action state machines to custom hooks — the prop surface explodes and the risk-to-reward is bad.
